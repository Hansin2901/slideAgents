# DATA MODEL
## Content
#### Text: Str 
will be provided as a pdf or a txt file or a md file

## Template slide
#### link: url
will be provided by user we will use this slides as our base and we will be editing here
#### presentationId: str

## Layout Explainations
#### shortDesc: str
it will be a short description of what is there on that layout
#### usecase: str
will be explaining how the layout should be used, for exampleif it was a title slide it would explain that. 
#### structureDescription: str
a textual description of the objects and their structrual setting of the layout. For example: there is one sub heading text box on the top left of the slide, the main body is split into 2 halves on one half you can put an image in the center, the other half is a text box for the main text body
#### usageTips: str (optional)
will give tips(llm based) if relevant and possible telling the agent something like, there are 3 main points you can fit on this slide if you need more points you will need to add those and restructure the left half of the slide to have n number of points and each of those text boxes should be equally spaced vertically
#### presentationId: str
#### chatId: str


## Slide Plan
#### slide number: int (1 indexed)
#### Layout name: str
it will learn that from the layout information
#### layoutObjectId: str
#### contentMap: dictionary{objectID(str):content(str)}
it will output the content that has to be put in each textbox in the layout, objectId will be provided to llm in the prompt and the content will be an xml string,the xml code the agent can indicate bold, italic, bullets etc.
The objectId's here will be generated according to a logic like textbox_{{number of textbox on the slide}} and then the slide number will be attached externally to make sure that the object ID is unique accross the whole presentation.
#### Reasoning: str
reasor for choosing that paticular layout for the provided  content
#### instructions: str
Instructions to the worker agent specific to that slide
#### isUpdateOperation:bool
if the slide was done and the human asked for an update this  flag would be true
#### presentationId: str
#### chatId: str

## todo list
#### task: str
#### time: time
#### chatId: str
#### presentationId: str
#### taskId: str
this will be handeled by an already implemented mcp
https://github.com/RegiByte/todo-list-mcp/blob/main/GUIDE.md

## Completed Slide Check
#### Slide number: int (1 indexed)
#### proposed changes: dictionary{objectID(str):content(str)}
same as the content in slide plan
the user  will be asked to go to the slide and check it out
#### updates: str
text description of updates made
#### aproval:Enum
can either be approved, rejected  or pending
#### presentationId: str
#### chatId: str
#### taskId: str

## Slide Make Command
#### Slide Plan: slide plan object
one generated by the planer llm
##### layoutDetails: layoutObject
the layout object will  basically be the layout explainations and the layout details from the slides api layout portion
#### slideIndex: int(1 indexed)
#### presentationId: str
#### chatId: str
#### taskId: str
#### slideData: slide object(optional)
if the planer sends an update slide request we will  also send in this data
