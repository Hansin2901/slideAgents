j# AI Memory - SlideAgents System

## Critical System Architecture

### Multi-Agent Design Pattern
- **Planner Agent**: Gemini 2.5 Pro - creates slide plans from content + layouts
- **Template Analyzer**: Gemini 2.0 Flash - one-time analysis per template  
- **Worker Agents**: SmolAgents (2-3 parallel) - execute slide creation via API
- **Queue System**: Task distribution with retry limits before human escalation

### Technology Stack Decisions
- **Package Manager**: uv (all Python operations)
- **Database**: SQLite (local development, AWS deployment later)
- **Framework**: LangGraph (orchestration) + SmolAgents (workers)
- **Interface**: CLI tool (Typer-based)
- **Validation**: Pydantic for all data models

## Key Data Model Patterns

### Object ID Generation Workflow (CRITICAL)
**Phase 1 - Template Analysis:**
1. **We generate temporary IDs** during layout analysis (`title_1`, `body_1`)
2. **We store temp IDs in database** with layout associations
3. **We provide temp IDs to planner in prompts** - planner doesn't generate them

**Phase 2 - Planning:**
4. **Planner uses our temp IDs** in content mapping
5. **Database tracks temp ID usage** by slides

**Phase 3 - Slide Creation:**
6. **We generate final IDs** using logic: `sld_{slide_number}_{temp_id}`
7. **We update database mapping** temp -> final
8. **Future operations use final IDs** from database

### ID Format Patterns
- **Temporary**: `title_1`, `body_1`, `image_1` (generated by us during layout analysis)
- **Final**: `sld_1_title_1`, `sld_2_body_1` (generated by us after slide creation)
- **Database tracks state**: TEMPORARY -> FINAL

### Content Mapping Structure
```
contentMap: {
  "objectId": "xml_formatted_content"
}
```
- XML format supports: `<bold>`, `<italic>`, `<ul><li>` for rich text
- Object IDs change from temp to final through database mapping

### Critical Workflow Sequence
1. OAuth -> Template Analysis (cached) -> Content Planning -> Task Queuing -> Parallel Execution -> Human Review -> Updates

## Essential Error Handling Requirements

### Retry Strategy
- Max retries configurable (3-5 attempts)
- Exponential backoff for API calls
- Human escalation after retry exhaustion
- Detailed error context for LLM guidance

### API Integration Points
- Google Slides API with rate limiting
- OAuth token refresh automation
- Todo-list MCP integration: https://github.com/RegiByte/todo-list-mcp

## Performance Targets
- Template analysis: <60 seconds
- Slide creation: 1 slide per 30 seconds
- System response: <5 seconds
- Memory usage: <2GB
- Concurrent presentations supported

## Security Requirements
- Credential encryption at rest
- No sensitive data logging
- HTTPS-only API communication
- Input validation and sanitization
- Audit trail for operations

## Deployment Architecture
- Start: CLI tool for local development
- Target: AWS containerized deployment
- Database migration path: SQLite -> cloud database
- Monitoring: Structured logging + metrics
